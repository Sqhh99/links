# 基于C++和WebRTC的多端实时音视频会议系统的设计与实现 — 开题报告

## 1. 毕业设计（论文）选题背景、研究意义（约400字）
近年来，5G/千兆宽带的普及和远程协同需求的爆发，使实时音视频会议从“可选”变为“刚需”。但在大并发、跨平台场景下，许多现有产品仍面临端到端延迟高、弱网自适应不足、屏幕共享卡顿、桌面客户端资源占用高等问题。开源生态中，WebRTC 提供了实时通信的标准化底座，但要落地到原生 C++/Qt 的多端产品，仍需解决信令设计、媒体流转发、设备抽象与跨平台渲染等工程难题。

本课题基于 C++17、Qt 6 和 LiveKit C++ SDK（WebRTC 核心）设计并实现多端实时音视频会议系统，支持多人音视频、屏幕共享、聊天、设备管理与现代化 UI。系统在客户端侧实现 GPU 加速渲染（OpenGL）、高效采集（摄像头/麦克风/屏幕）与数据通道即时通讯；在服务端侧通过 LiveKit SFU 与 Go 信令服务提供低延迟分发与鉴权。研究成果可为企业协同、远程教育、远程诊疗等场景提供可复用的技术路径，也为 C++ 在实时通信领域的工程化落地提供参考，对提升国产化 RTC 技术自主可控能力具有实践意义。

## 2. 国内外研究现状和发展趋势（约600字）
**国内现状与趋势。** 国产会议产品（腾讯会议、钉钉、飞书）已形成成熟商业方案，但开源、可定制的原生客户端方案相对匮乏。学术与工程侧研究集中在：① 基于 WebRTC 的 SFU 架构与带宽自适应；② 视频编解码优化与硬件加速（H.264/H.265/AV1）；③ 弱网抗抖动与自适应码率；④ 屏幕共享链路的清晰度/延迟平衡。已有工作多聚焦浏览器或 Electron，针对纯 C++/Qt 的跨平台桌面实现与低资源占用的研究仍不足。声网、融云等商用 SDK 在生态和运维上成熟，但底层完全开源的可控方案仍需补位。

**国外现状与趋势.** WebRTC 由 Google 开源并成为 W3C/IETF 标准，社区围绕 TURN/STUN、拥塞控制、回声消除、前后处理等持续迭代。LiveKit 作为新一代开源 SFU 平台，提供跨语言 SDK（含 C++/Rust FFI）和完整信令/房间/权限模型，成为构建自托管 RTC 服务的热门方案。研究热点包括：① 5G/边缘计算环境下的端到端时延优化；② 基于 AI 的回声消除、降噪、虚拟背景与带宽预测；③ 多编码器协同与 SVC/Simulcast 选择；④ WebAssembly 与桌面原生的混合形态，降低跨平台成本。

**发展趋势.** 短期聚焦“低延迟+高画质+弱网鲁棒”，中期走向“AI 增强”（智能降噪、语音增强、背景抠图、超分辨率）、“端到端加密”与“边缘协同”，长期将强调开源可控、自托管能力与隐私合规（GDPR、本地政企标准），并在多设备形态（PC/移动/嵌入式）上统一开发体验。

## 3. 主要研究内容和研究方法（≥600字）
**功能模块设计（客户端）**
- 用户登录与鉴权：房间创建/加入、JWT Token 获取，基于 Go 信令服务 REST API。
- 会议管理（ConferenceManager）：房间连接、参与者状态同步、轨道发布/订阅、数据通道聊天、自动重连与事件分发。
- 摄像头采集（CameraCapturer）：Qt Multimedia 设备枚举，帧率控制（30fps），向 LiveKit VideoSource 送帧。
- 麦克风采集（MicrophoneCapturer）：48kHz/16bit PCM 采集，状态监测与错误处理，向 LiveKit AudioSource 送帧。
- 屏幕共享（ScreenCapturer）：Windows 下 DXGI Desktop Duplication/WinRT Graphics Capture，支持全屏/窗口选择，对话框筛选窗口。
- 视频渲染（GLVideoWidget）：OpenGL ES 2.0 着色器渲染、纹理流更新、镜像显示、昵称/状态叠加，降低 CPU 占用。
- 即时通讯（ChatPanel）：LiveKit DataChannel 可靠消息，文本发送/展示/标记本地或远端。
- 参会者管理（ParticipantItem）：参会者列表、音视频状态、主画面切换。
- 设置与持久化（SettingsDialog/Settings）：设备选择、API/信令地址、用户偏好，基于 QSettings 本地存储。

**服务端与信令**
- 信令/鉴权服务：Go + Gorilla Mux + LiveKit Server SDK，提供 Token 颁发、房间管理、健康检查，支持 HTTPS 与 CORS。
- 媒体转发：依托 LiveKit SFU，实现多方选择性转发、拥塞控制与带宽自适应。

**研究方法与技术路线**
1) 体系架构：C++/Qt 客户端（UI、采集、渲染、控制） + LiveKit SFU + Go 信令服务；模块化分层（core/ui/utils/res）。
2) 关键技术点：
   - WebRTC/LiveKit 轨道发布订阅、DataChannel 即时通讯。
   - 采集链路：摄像头/麦克风/屏幕采集到 LiveKit 源的格式转换与节流。
   - 渲染链路：OpenGL 纹理更新、帧率控制、状态叠加。
   - 弱网与稳定性：带宽自适应依赖 LiveKit/RTC 堆栈；本地侧增加错误处理与重连策略。
3) 开发与测试：
   - 构建：CMake + Qt 6，集成 Rust FFI 的 LiveKit C++ SDK。
   - 功能测试：多人入会、轨道发布/订阅、屏幕共享、聊天、设备切换。
   - 性能测试：端到端时延、帧率、CPU/GPU 占用，弱网（限带宽/抖动）场景验证。
   - 兼容性测试：Windows 主平台，预留 macOS/Linux 适配点。
4) 迭代与优化：按模块交付，先“可用”，后“好用”与“稳健”，逐步完善 UI 与异常恢复。
